## 运行环境

- 所有命令都从项目根目录运行。
- 禁止使用 Python 或自编脚本提取、推断、修复字段。只允许 Read 直接查看 PDF，以及使用下文明确给出的 `pdftotext` 命令辅助读取。

## 错误位置

从 `报销工作文件/invoice_errors_raw.json` 读取错误列表。该文件由 `check_invoice_errors.py` 生成，格式如下：

```json
{
  "has_error": true,
  "error_count": 2,
  "errors": [
    {
      "文件名": "example.pdf",
      "字段": "项目列表.0.单价",
      "当前值": "ERROR",
      "错误类型": "ERROR"
    }
  ]
}
```

## 修复要求

### 范围限制
- **只修正 `报销工作文件/invoice_errors_raw.json` 的 `errors[]` 中列出的字段**。其他任何字段（包括非 ERROR 的 `项目名称`、`价税合计金额`、`销售方名称`、`发票号码`、`开票时间` 等）一律不碰。
- 不要添加、删除或修改 `项目列表` 中的条目总数，只修正 `单价` 的值。
- 保持现有 JSON 结构，不要引入新键。
- 如果 `报销工作文件/invoice_errors_raw.json` 不存在、不是有效 JSON、`error_count` 为 0，或其中某个错误无法从 PDF/行程单可靠修复，立即停止并向主流程报告，不要猜测。

### 从 PDF 提取正确值

如果是打车发票，需同时看发票 PDF 和对应的行程单 PDF 来提取正确的单价。

```bash
pdftotext -layout invoices/<文件名.pdf> -
pdftotext -layout invoices/<对应行程单文件名.pdf> -
```

- 发票上每行 `项目名称` 对应一次行程，`单价` 为该行程金额。
- 行程单上列出了每次行程的明细金额，用行程单中的金额来修正发票上对应行程的 `单价`。
- 如果行程单中的次数和发票条目数不对应，检查发票总金额验证。

### 输出

将修正值**直接写入** `报销工作文件/invoice_fixes.json`，以 `文件名` 为键：

```json
{
  "example.pdf": {
    "购买方税号": "12100000470095016Q",
    "价税合计金额": 39.04,
    "项目列表.0.项目名称": "交通运输服务",
    "项目列表.0.单价": 20.49,
    "开票时间": [
      {
        "年": 2026,
        "月": 7,
        "日": 1
      }
    ],
    "发票号码状态": "正常"
  }
}
```

输出结构必须是“发票文件名 → 需要修正的字段 → 正确值”：

- 外层键必须原样使用错误列表中的 `文件名`。
- 内层键必须原样使用错误列表中的 `字段`，不得改写或保留示例占位符。
- 内层值必须替换为从对应 PDF 或行程单可靠提取的正确值。
- `字段` 已经是 `apply_invoice_fixes.py` 支持的路径格式。嵌套字段使用 `项目列表.0.单价`、`项目列表.0.项目名称` 等路径；顶层字段直接使用 `开票时间`、`发票号码状态` 等字段名。
- 最终文件只能包含错误列表中需要修正的字段，示例中的其他字段不得照搬。

严格遵守以下 JSON 类型：

- `价税合计金额` 以及路径以 `.单价` 结尾的字段必须是非负 JSON number，禁止加引号。
- `开票时间` 必须是仅包含日期对象的 JSON array；每个日期对象必须包含 `年`、`月`、`日`，且三者均为 JSON integer。
- 税号、名称、发票号码、项目名称和状态字段必须是 JSON string。
- `发票号码状态` 只能是 `正常` 或 `需人工校验`。
- 写入后重新读取 `invoice_fixes.json`，逐字段确认键名来自错误列表且值符合上述类型；任何字段无法满足时立即停止并报告，不要输出不完整或类型不正确的修复文件。

`项目列表` 必须包含原始所有条目（不增不减），只修改 `单价` 的值。只写入需要修改的发票。

输出一条简短摘要（如修复了 N 张发票中的 M 个字段）。
